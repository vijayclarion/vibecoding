## 🏛️ CONSTITUTION FILES

The Constitution is the **source of truth** for your project. Copilot reads these first.

### 1️⃣ `specs/mission.md`

**What it is:** Product intent. What you're building and for whom.

**When to create:** At project start (day 1).

**Who writes it:** You + Copilot (conversational).

**Size:** 500-800 words.

**Sections:**

```markdown
# Mission: [ProjectName]

## The Problem
[2-3 sentences about the pain you're solving. Include numbers if possible.]

## Our Solution
[1 paragraph. What does the product do, at a high level? NO tech details.]

## Target Audience
[List primary + secondary users. Who actually uses this?]

## Core Principles (ranked)
1. [Most important principle]
2. [Second most important]
3. [Third]
4. [Fourth]

## Success Metrics
[Measurable, time-bound. E.g., "Reduce reconciliation from 4h to <15min"]

## Non-Goals
[What we explicitly will NOT do. Just as important as what we will.]

## Constraints
[Regulatory, deadline, budget constraints.]

## Glossary
[Domain terms — so everyone uses the same vocabulary.]
```

**What Copilot uses this for:** Understanding project context, asking if scope is reasonable, checking if decisions align with vision.

**Example:**
```
Read specs/mission.md. Does the feature I'm about to build align 
with "Correctness over speed"? If not, should we reconsider?
```

### 2️⃣ `specs/tech-stack.md`

**What it is:** Architectural and technical decisions. HOW you'll build.

**When to create:** Day 1-2, after mission.md.

**Who writes it:** You + Copilot (conversational).

**Size:** 1000-1500 words.

**Sections:**

```markdown
# Tech Stack: [ProjectName]

## Languages & Frameworks
[Node.js 20, Express.js, etc. — specific versions.]

## Key Libraries
[Table of: Library | Version | Why chosen]

## Folder Structure
[ASCII diagram of src/, tests/, etc.]

## Architectural Patterns
[Monolith? Microservices? Event-driven? REST? GraphQL?]

## Data Model (High-Level)
[ASCII diagram: Customer --< Order --< LineItem]

## Request Lifecycle
[Trace one key endpoint through the entire system.]

## Cross-Cutting Concerns
[Auth, logging, error handling, validation, observability, config, secrets]

## Testing Strategy
[Unit + integration + E2E? Frameworks?]

## Quality Gates
[Linting, type-checking, coverage targets, CI/CD checks]

## Non-Functional Requirements
[P95 latency, availability, throughput, RTO/RPO]

## Conventions
[5-10 key rules teams must follow always]
```

**What Copilot uses this for:** Code generation style, library choices, error handling patterns, testing approach, naming conventions.

**Example:**
```
You said we use "Zod for validation" in tech-stack.md.
This code uses manual validation instead. Shall I rewrite to use Zod?
```

### 3️⃣ `specs/roadmap.md`

**What it is:** Phased feature plan. What's next.

**When to create:** Day 2, after mission + tech-stack.

**Who writes it:** You + Copilot (conversational).

**Size:** 300-600 words (grows as you add phases).

**Sections:**

```markdown
# Roadmap: [ProjectName]

## Phase 0 — Foundation
- [ ] F0.1 — Project scaffold + CI/CD
- [ ] F0.2 — Database schema
- [ ] F0.3 — Health check endpoint
- [ ] F0.4 — Error handling + logging baseline

**Exit criteria:** [What "done" looks like for Phase 0]

## Phase 1 — MVP Core
- [ ] F1.1 — First valuable feature (thinnest slice)
- [ ] F1.2 — Next
- [ ] F1.3 — Next
- [ ] F1.4 — Tests + hardening

**Exit criteria:** [What MVP means for you]

## Phase 2 — MVP Polish
- [ ] F2.1 — Auth
- [ ] F2.2 — Documentation
- [ ] F2.3 — Monitoring

## Phase 3 — v1.0
- [...]

## Deferred / Parking Lot
[Features we discussed but won't build soon — with reason]

## Revision History
[Track changes]
```

**What Copilot uses this for:** Scope management, feature breakdown validation, priority checking, understanding what's next.

**Example:**
```
I'm about to implement payment processing.
Looking at roadmap.md, that's F2.1, but we haven't finished Phase 1 yet.
Should I focus on F1.4 (tests) instead?
```

### 4️⃣ `.github/copilot-instructions.md` (or `COPILOT.md`)

**What it is:** Rules Copilot must follow. Your enforcement mechanism.

**When to create:** Day 2-3 (after understanding your conventions).

**Who writes it:** You (with examples from tech-stack.md).

**Size:** 400-700 words.

**Sections:**

```markdown
# GitHub Copilot Instructions

You are building [ProjectName], a [tech stack description].

## Before You Generate Anything
Read these files in this order:
1. /specs/mission.md
2. /specs/tech-stack.md
3. /specs/roadmap.md
4. /specs/features/[current-feature]/spec.md (if working on a feature)

## Core Rules (Always Apply)

### Coding Conventions
- Language: [with strict mode settings]
- Framework: [and version]
- Validation: [library - e.g., Zod]
- Error handling: [return this exact shape]
- Logging: [JSON with these fields]
- Testing: [framework and patterns]
- **No:** [list things Copilot should never do]

### Architecture Rules
- Routes are thin (delegate to services)
- Services contain business logic
- Models handle persistence
- Middleware handles cross-cutting concerns

### Error Response Format
[Copy the exact JSON structure from tech-stack.md]

### Logging Format
[Copy the exact structure from tech-stack.md]

### Testing Requirements
- Every route: ≥3 tests (happy path + 2 error paths)
- Test framework: [specific framework + patterns]
- Minimum coverage: 85%

## When You're Stuck
- Use [NEEDS CLARIFICATION] to flag uncertainty
- Stop and ask rather than guess
- Reject code that violates above rules

## Definition of Done
Code is complete only when:
- [ ] Compiles with zero errors/warnings
- [ ] Tests pass
- [ ] Lint passes
- [ ] Type check passes
- [ ] New code ≥85% covered
- [ ] Follows all conventions above
```

**What Copilot uses this for:** Validation, error checking, style enforcement, preventing bad patterns.

**Example:**
```
You generated an error response with shape { success: false, message: "..." }
But .github/copilot-instructions.md says all errors use ProblemDetails.
Let me fix that.
```

### 5️⃣ `specs/features/[NNN]-[name]/spec.md`

**What it is:** Feature intent. What this feature does, when it's done.

**When to create:** Before any code, for each feature.

**Who writes it:** You + Copilot (conversational, iterative).

**Size:** 400-600 words.

**Key sections:**

```markdown
# Feature: [Feature Name] (F[N.N])

## User Story
As a **[role]**, I want to **[action]**, so that **[benefit]**.

## Acceptance Criteria
- [ ] **AC1** — Given [precondition], when [action], then [outcome]
- [ ] **AC2** — ...
- [ ] **AC3** — Edge case: ...
- [ ] **AC4** — Error case: ...

## Requirements

### Functional
- [explicit requirement]
- [explicit requirement]

### Non-Functional
- Performance: <target>
- Security: <requirement>
- Logging: <what must be logged>

### Out of Scope
- [explicitly NOT doing]

## Dependencies
- Prerequisite features: [list]
- External services: [list]

## Validation Scorecard
- [ ] Happy path works
- [ ] Error cases handled
- [ ] Tests pass
- [ ] Logs correct
- [ ] Performance OK
```

**What Copilot uses this for:** Understanding requirements, asking clarifying questions, validating generated code against ACs.

**Example:**
```
AC3 says "Empty line_items → 400 Bad Request with field-level error".
Your generated code returns 422. Should I change it to 400?
```

### 6️⃣ `specs/features/[NNN]-[name]/plan.md`

**What it is:** Technical approach. How you'll build the feature.

**When to create:** After spec.md is approved.

**Who writes it:** You + Copilot (conversational).

**Size:** 500-800 words.

**Key sections:**

```markdown
# Plan: [Feature Name]

## Approach
[1-2 paragraphs explaining WHY this approach over alternatives]

## Files to Create
| Path | Purpose |
|---|---|
| src/types/order.ts | TypeScript types |
| src/routes/orders.ts | HTTP endpoints |
| tests/... | Tests |

## Files to Modify
| Path | Change |
|---|---|
| src/app.ts | Register router |
| package.json | New deps? |

## Data Model Changes
[SQL or ORM code snippet]

## Request/Response Contract
[Exact JSON shapes with all status codes]

## Risks & Mitigations
| Risk | Likelihood | Impact | Fix |
|---|---|---|---|
| [Risk 1] | [Likelihood] | [Impact] | [Mitigation] |

## Alternatives Considered
- [Considered but rejected]
```

**What Copilot uses this for:** File generation, API contract enforcement, error handling patterns, architecture validation.

**Example:**
```
The plan says the response should include { id, status, createdAt }.
Your code returns { id, status, timestamp }. 
Should I update to use "createdAt" instead?
```

### 7️⃣ `specs/features/[NNN]-[name]/tasks.md`

**What it is:** Atomic work units. Step-by-step breakdown.

**When to create:** After plan.md is approved.

**Who writes it:** You + Copilot (conversational).

**Size:** 300-500 words (list of tasks).

**Format:**

```markdown
# Tasks: [Feature Name]

## Setup
- [ ] **T1** — Create branch: git checkout -b feature/NNN-name
- [ ] **T2** — Run tests to confirm they work: npm test

## Data Layer
- [ ] **T3** — Create src/types/order.ts with interfaces
- [ ] **T4** — Create migration: [migration command]
- [ ] **T5** — Run migration: [command]

## Service Layer
- [ ] **T6** — Create src/services/orderService.ts
- [ ] **T7** — Unit tests for orderService
  - Happy path
  - Error case 1
  - Error case 2

## API Layer
- [ ] **T8** — Create src/routes/orders.ts
- [ ] **T9** — Wire up in src/app.ts
- [ ] **T10** — Integration test: happy path
- [ ] **T11** — Integration test: error cases
- [ ] **T12** — Integration test: validation errors

## Quality Gates
- [ ] **T13** — npm run lint
- [ ] **T14** — npm run typecheck
- [ ] **T15** — npm test
- [ ] **T16** — Coverage check

## Commit & Merge
- [ ] **T17** — git commit
- [ ] **T18** — Open PR
- [ ] **T19** — Merge
- [ ] **T20** — Update roadmap.md
```

**What Copilot uses this for:** Task-by-task code generation, scope management, preventing scope creep.

**Example:**
```
You asked me to implement T3 (create types).
I can do that without implementing the whole feature.
Should I stop after the types are done?
```

### 8️⃣ `specs/features/[NNN]-[name]/validation.md`

**What it is:** How to verify the feature works.

**When to create:** After tasks.md is approved.

**Who writes it:** You (before implementation, or reviewed during).

**Size:** 400-600 words.

**Key sections:**

```markdown
# Validation: [Feature Name]

## Automated Checks
```bash
npm test                 # All green?
npm run lint            # Zero warnings?
npm run typecheck       # Zero errors?
npm run coverage        # ≥85%?
```

## Acceptance Criteria Verification
| AC | Test | Result |
|---|---|---|
| AC1 | test name | ☐ |
| AC2 | test name | ☐ |

## Manual Smoke Test
[curl commands or step-by-step instructions]

## Non-Functional Checks
- [ ] P95 latency < target
- [ ] No memory leaks
- [ ] Logs have correlation id
- [ ] No regressions

## Sign-off
- [ ] Dev: ready for review
- [ ] Tests: all green
- [ ] Copilot: code quality OK
- [ ] Merge to main
```

**What Copilot uses this for:** Validation verification, test completeness checking.

## 📖 CONTINUOUS LEARNING FILE

### 9️⃣ `specs/LessonsLearned.md`

**What it is:** Growing log of surprises, gotchas, patterns discovered.

**When to create:** During or after first feature.

**Who writes it:** You (with Copilot's help proposing entries).

**Size:** Grows over time (starts empty).

**Format:**

```markdown
# Lessons Learned

## YYYY-MM-DD — F1.1 — [Title]

**Context:** What we were trying to do.

**What happened:** What actually happened.

**Root cause:** Why it happened.

**Fix:** What we did about it.

**Rule going forward:**
- [Rule 1]
- [Promoted to: tech-stack.md if project-wide]

**References:** Commit hash, test name, link to discussion.

---
```

**Example:**

```
## 2024-04-22 — F1.1 — Race condition in idempotency check

**Context:** Implementing idempotent order creation with Idempotency-Key header.

**What happened:** Two simultaneous requests with same key both created orders.

**Root cause:** "Check if key exists" then "insert order" wasn't atomic.

**Fix:** Used unique database constraint; catch violation on insert.

**Rule going forward:**
- Idempotency MUST use DB constraint, never application-level check.
- Promoted to: specs/tech-stack.md § Cross-Cutting Concerns
```

## 🎯 SUMMARY: What Copilot Reads & When

| File | When Copilot reads it | Why |
|---|---|---|
| `mission.md` | Every conversation start | Understands project context |
| `tech-stack.md` | Every code generation | Follows conventions + libraries |
| `roadmap.md` | When scope is ambiguous | Checks if feature is in scope |
| `copilot-instructions.md` | Before generating code | Enforces project rules |
| `features/NNN/spec.md` | When implementing feature | Knows what to build |
| `features/NNN/plan.md` | When designing code | Knows HOW to build it |
| `features/NNN/tasks.md` | When implementing per-task | Knows exact scope of task |
| `features/NNN/validation.md` | When verifying results | Knows how to check if it works |

## 🚀 Quick Start: Files to Create First

**Day 1:**
1. Create `specs/` folder
2. Create `specs/mission.md` ← ask Copilot to help
3. Create `specs/tech-stack.md` ← ask Copilot to help
4. Create `specs/roadmap.md` ← ask Copilot to help
5. Create `.github/copilot-instructions.md` ← write yourself

**Day 2:**
1. Create `specs/features/001-first-feature/` folder
2. Create `spec.md` ← ask Copilot to help
3. Create `plan.md` ← ask Copilot to help
4. Create `tasks.md` ← ask Copilot to help
5. Create `validation.md` ← write yourself

**Days 3+:**
For each task in `tasks.md`:
1. Ask Copilot to implement task T1
2. Review code
3. Run tests
4. Approve, move to T2

---

## 💡 Pro Tips

- **Commit all `.md` files.** They're part of your codebase.
- **Keep specs in sync with code.** If code diverges from spec, update spec.
- **Reference specs in commits.** "feat: F1.1 create order — see specs/features/001-create-order/"
- **Use `@file` in Copilot prompts.** Tells Copilot which file to read: "@spec tell me AC3"
- **Ask Copilot to review specs for clarity.** "Is this spec clear enough that a new dev could implement it?"

---

**Remember:** Better specs = less back-and-forth with Copilot = faster shipping.
